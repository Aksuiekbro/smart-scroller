## Spacing Scale

The existing options/options.css uses an 8-point-derived scale. These values are locked — do not introduce new spacing tokens.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Chip gap, icon-to-label gap, `.ss-eyebrow` bottom margin |
| sm | 8px | Row gap (`.row { gap: 10px }` rounds to this), chip padding, list item gap |
| md | 16px | Section margin-bottom, horizontal page padding |
| lg | 20px | Card padding (`.card { padding: 20px }`) |
| xl | 24px | Wrap top padding, footer margin-top |
| 2xl | 28px | Header margin-bottom (`.hdr { margin-bottom: 28px }`) |
| 3xl | 40px | Wrap top padding |

Exceptions:
- `.wrap` uses `padding: 40px 24px 80px` — asymmetric vertical padding is intentional (space for the footer).
- Tap targets on iOS must be at minimum 44×44px. Buttons with `padding: 7px 13px` on 13px text achieve ~27px height — pair with `min-height: 44px` on all interactive elements in the diagnostic page and options additions.
- `.card { border-radius: 14px }` and `.topic { border-radius: 12px }` — use 14px for new `.card` instances.

## Typography

### New Phase 2 declarations (in-scope for the 2-weight / 4-size constraint)

Phase 2 introduces new classes (`.result-row`, `.result-icon`, `.result-label`, `.result-value`, `.result-threshold`, `.spinner`) and extends the existing options page with new sections. All new code uses exactly **two weights** and **four sizes**:

| Role | Size | Weight | Line Height | CSS Class / Usage |
|------|------|--------|-------------|-------------------|
| Small / hint / threshold | 12px | 400 (regular) | 1.4 | `.hint`, `.status`, `.result-threshold` — threshold annotation reuses the existing 12px `.hint` size to keep the type scale minimal |
| Label / UI chrome | 13px | 400 (regular) | 1.3 | `.result-label` metric name |
| Body / heading | 14px | 400 (regular) body / 600 (semibold) h2 | 1.45 body / 1.2 h2 | Default `body` font-size; `h2` section titles — h2 visual emphasis comes from weight (600) and section spacing, not size; keeps the type scale to 4 sizes (12/13/14/24) |
| Page heading | 24px | 600 (semibold) | 1.0 (letter-spacing: -0.01em) | `h1` in `.hdr` |

**Declared weights for new Phase 2 code: 400 (regular) and 600 (semibold). Exactly 2.**

**Declared sizes for new Phase 2 code: 12px / 13px / 14px / 24px. Exactly 4.**

Note on `h2`: The locked `options/options.css` source file declares `h2 { font-size: 15px; font-weight: 600 }`. Phase 2 new sections in the options page inherit this existing rule — it is NOT a new declaration by this UI-SPEC. The diagnostic page (`diagnostic/diagnostic.css`) declares `h2 { font-size: 14px; font-weight: 600 }` as its own new rule (within the 4-size budget). The 15px size in options.css is locked by the existing source and is not counted in Phase 2's type scale.

|-------|----------------|--------|------|------------|
| Idle (no run yet) | `--card` (no tint) | `--border` | — | `--muted` |

### 60/30/10 split

- **60% dominant** — `--bg` (`#fafafb` / `#0f1014`): page background, `.wrap`
- **30% secondary** — `--card` (`#fff` / `#17181d`): all `.card` blocks, result table background
- **10% accent** — `--accent` (`#0a7cff`): reserved for: spinner animation, active toggle state, focus ring on inputs, "Load embedder" primary button on the diagnostic page

`--danger` (`#d33`) is reserved for destructive actions only: "Remove topic" button, and any destructive confirmation copy. Do NOT use it for diagnostic fail state (use `#ff3b30` per above).

## Interaction Contracts

### Diagnostic page — "Load embedder" button

1. User taps "Load embedder."
2. Button text becomes "Loading…"; button is `disabled` (prevents double-run).
3. `.spinner` appears in the first `.result-row` `.result-icon` column.
4. Embedder loads; measurements collected.
5. Within 300ms of pipeline completing: result rows populate with measured values.
6. Each row gets `.result-row--pass` or `.result-row--fail` based on threshold comparison.
7. Summary row (final row in the table) shows overall pass/fail.
8. Button text becomes "Re-test embedder"; button re-enabled.
9. "Copy to TRANSFORMERS-DECISION.md" button becomes enabled (was disabled before first run).
10. If an exception is thrown: single `.result-row--fail` row with `Error: {error.message}` in `.result-label`. All other rows hidden or show `—`.

### Diagnostic page — "Copy to TRANSFORMERS-DECISION.md" button

1. User taps "Copy to TRANSFORMERS-DECISION.md."
2. `navigator.clipboard.writeText(markdownBlock)` called.
3. Button text becomes "Copied!" immediately.
4. After 1 500ms, button text reverts to "Copy to TRANSFORMERS-DECISION.md."
5. If clipboard API unavailable (should not happen on Orion, but guard): show `alert("Copy failed. See console.")` and `console.log(markdownBlock)`.

Markdown block format (prescriptive):

```
## Chosen: v{version}

### Measurements (iPhone Orion {YYYY-MM-DD})
| Metric | Value | Threshold | Pass |
|--------|-------|-----------|------|
| Cold-start time | {N} ms | ≤ 8 000 ms | {✓/✗} |
| Peak memory | {N} MB | ≤ 250 MB | {✓/✗} |
| Vector dimensions | {N} | = 384 | {✓/✗} |

### Notes
{error.message if any, or "All checks passed."}
```

### Options page — Export settings

1. User taps "Export settings."
2. JS reads `chrome.storage.sync` (topics, enabled, sites, pauseUntil — D-12; stats excluded).
3. `JSON.stringify(payload, null, 2)` → Blob → object URL → programmatic `<a download>` click.
4. Filename: `smartscroller-settings-YYYY-MM-DD.json`.
5. No confirmation dialog needed (non-destructive).
6. Inline status (`.status`, `aria-live="polite"`) shows "Exported." for 2 000ms then clears.

### Options page — Import settings (file picker, primary path — D-13)

1. User taps "Choose file…" label (which activates `<input type="file" accept="application/json,.json">`; `multiple` is NOT set).
2. iOS Files picker opens.
3. User selects a `.json` file.
4. `FileReader.readAsText(file)` → `JSON.parse()`.
5. If parse fails → show inline hint above file input: "Could not read that file. Make sure it's a SmartScroller export." Stop; do not proceed to confirm.
6. Validate shape: object must have `topics` (array), `enabled` (boolean), `sites` (object). If invalid shape → show "That file doesn't look like a SmartScroller backup." Stop.
7. Count N = current topics from storage, M = imported topics count.
8. `confirm("⚠ This will replace your N existing topic(s) with M from the file. Continue?")` (native dialog — works on Orion iOS).
9. If user cancels: abort, no storage write.
10. If user confirms: `chrome.storage.sync.set(importedPayload)` (replace-all, D-14).
11. Inline status: "Imported M topic(s). Settings replaced." for 2 000ms.
12. `renderTopics()` called to refresh the Topics section.

### Options page — Import settings (paste textarea, fallback — D-13)

Same flow as file picker from step 4 onward, but:
- Source is textarea `.value` instead of `FileReader` output.
- Inline hint for errors appears above the "Import from text" button, not above the file input.
- Textarea clears on successful import.

### Button states summary

| Button | Enabled state | Disabled state | Notes |
|--------|--------------|----------------|-------|
| Load embedder | Default | During run (disabled attr) | Text changes to "Loading…" while disabled |
| Re-test embedder | After first run | During run | Same button; label swaps |
| Copy to TRANSFORMERS-DECISION.md | After first run | Before first run (disabled attr) | Flash "Copied!" for 1 500ms |
| Export settings | Always | — | Non-destructive, no disable needed |
| Choose file… | Always | — | File picker trigger |
| Import from text | When textarea non-empty | When textarea is empty | JS toggles disabled |

## Accessibility and iOS Orion Considerations

All new interactive elements must meet these requirements.

| Requirement | Implementation |
|-------------|----------------|
| Tap targets ≥ 44×44px | All buttons get `min-height: 44px` in CSS. The file-picker `<label>` also gets `min-height: 44px`. |
| `aria-live="polite"` | All inline status elements (import status, export status, diagnostic run status) use `aria-live="polite"`. Same pattern as `options/options.html:16`. |
| File picker | `<input type="file" accept="application/json,.json">` — no `multiple` attribute. Label is the visual trigger. |
| Confirm dialog | Native `confirm()` — works on iOS Safari / Orion out of the box. No custom modal needed. |
| Focus states | Existing `options/options.css` uses `:focus { outline: none; border-bottom-color: var(--accent); }` for text inputs. Buttons rely on browser default focus ring — acceptable for a personal tool. |
| Disabled buttons | Use `disabled` attribute (not just `pointer-events: none`) so assistive tech knows the button is unavailable. |
| `color-scheme: light dark` | Already declared in options/options.css `:root`. Diagnostic page must also declare it (or inherit via linked stylesheet). |
| Semantic HTML | `<section class="card">` preferred over bare `<div class="card">` for new sections in options page. Diagnostic page uses `<main class="wrap">`. |

## Error States

| Surface | Trigger | Treatment |
|---------|---------|-----------|
| Diagnostic — embedder exception | `try/catch` catches any error | Single `.result-row--fail` row; `.result-label` = `Error: {error.message}`; all metric rows hidden or show `—`. Summary row: `✗ Check(s) failed — see highlighted rows`. |
| Diagnostic — individual threshold breach | Measured value exceeds threshold | That `.result-row` gets `.result-row--fail`; `.result-threshold` appends `(failed: {value} > {threshold})`. Other rows unaffected. |
| Import — malformed JSON | `JSON.parse()` throws | Inline `.hint` above the file-picker row: `Could not read that file. Make sure it's a SmartScroller export.` Red tint: `color: #ff3b30`. Do NOT call `confirm()`. |
| Import — wrong shape | Object missing required keys | Inline `.hint`: `That file doesn't look like a SmartScroller backup.` Same red tint. |
| Import — user cancels confirm | `confirm()` returns false | Silent abort — no error message shown. Textarea/file input left as-is. |
